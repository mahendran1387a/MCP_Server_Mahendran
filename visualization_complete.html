üìä Complete Request Flow (12 Steps)

  1. User asks question in browser
  2. Flask receives HTTP POST to /api/query
  3. AsyncClientManager routes to MCP client
  4. MCP Client sends to Ollama with tool descriptions
  5. Ollama analyzes and decides which tool to use
  6. MCP Client calls MCP server with tool request
  7. MCP Server executes appropriate tool
  8. RAG Tool (if used) calls HTTP API ‚Üí web server ‚Üí ChromaDB
  9. Other Tools execute directly (calculator, weather, etc.)
  10. Tool Result returns to MCP client
  11. Ollama formats result into natural language
  12. Response bubbles back through all layers to browser

üîÑ Before/After: Database Locking Solution

‚ùå BEFORE (Problematic)
Browser ‚Üí Web Server ‚îú‚îÄ‚îÄ Opens ChromaDB ‚úÖ ‚îî‚îÄ‚îÄ Routes to MCP Client ‚îî‚îÄ‚îÄ MCP Server ‚îî‚îÄ‚îÄ ‚ùå TRIES to open ChromaDB ‚ö†Ô∏è DATABASE LOCKED! üí• Connection closed error

Problems:

  • Both web_server.py and mcp_server.py try to open ChromaDB
  • ChromaDB uses file-based locking
  • Second connection attempt fails
  • MCP server subprocess crashes
  • Error: "Connection closed"
  • Misdiagnosed as "Ollama is not running"

Code Issue:

# mcp_server.py (OLD - BROKEN)
class MCPServer:
    def __init__(self):
        self.server = Server("langchain-ollama-mcp")
        self.rag_system = RAGSystem()  # ‚ùå Opens ChromaDB
        self.setup_handlers()
‚úÖ AFTER (Fixed)
Browser ‚Üí Web Server ‚îú‚îÄ‚îÄ Opens ChromaDB ‚úÖ (ONLY connection) ‚îú‚îÄ‚îÄ ‚úÖ Provides /api/rag/query endpoint ‚îî‚îÄ‚îÄ Routes to MCP Client ‚îî‚îÄ‚îÄ MCP Server ‚îî‚îÄ‚îÄ ‚úÖ Uses HTTP API for RAG POST /api/rag/query üéâ No database access!

Solutions:

  • ‚úÖ Only web_server.py opens ChromaDB
  • ‚úÖ MCP server uses HTTP API for RAG queries
  • ‚úÖ Added /api/rag/query POST endpoint
  • ‚úÖ No more database locking conflicts
  • ‚úÖ No more "Connection closed" errors
  • ‚úÖ System works flawlessly!

Code Fix:

# mcp_server.py (NEW - FIXED)
class MCPServer:
    def __init__(self):
        self.server = Server("langchain-ollama-mcp")
        # Don't initialize RAG system here
        # RAG queries use HTTP API instead
        self.setup_handlers()

async def rag_query_tool(self, arguments: dict):
    # Make HTTP request to web server
    req = urllib.request.Request(
        "http://localhost:5000/api/rag/query",
        data=json.dumps(arguments).encode('utf-8')
    )
    response = urllib.request.urlopen(req)
    return response

üí° Key Takeaway

By using an HTTP API architecture, we solved the database locking problem while maintaining full RAG functionality. This is a production-ready solution that scales well and prevents concurrent access issues.

‚ö†Ô∏è Common Errors & Solutions

‚ùå Error: "Ollama is not running" ‚ñº

Symptoms:

  • Browser shows "Ollama is not running" message
  • But Ollama IS running (port 11434 is bound)
  • Server logs show "Connection closed"

Root Cause:

This error message is misleading! The real problem is ChromaDB database locking, not Ollama.

‚úÖ Solution:

  1. Run diagnostic: python test_ollama.py
  2. If Ollama test passes, it's not Ollama!
  3. Check if you're using the fixed version with HTTP API for RAG
  4. Ensure mcp_server.py doesn't open ChromaDB directly
‚ùå Error: "Connection closed" ‚ñº

Symptoms:

  • MCP server subprocess crashes
  • Error in logs: "mcp.shared.exceptions.McpError: Connection closed"
  • Happens during MCP client initialization

Root Cause:

MCP server tries to open ChromaDB while web server already has it open. File locking prevents second connection.

‚úÖ Solution:

  1. Use the fixed architecture with HTTP API
  2. Verify mcp_server.py uses urllib.request for RAG
  3. Check web_server.py has /api/rag/query endpoint
  4. Restart the web server
‚ùå Error: Ollama not accessible ‚ñº

Symptoms:

  • Connection refused to localhost:11434
  • Actual Ollama connectivity issue

‚úÖ Solution:

  1. Windows: Run ollama serve in PowerShell
  2. Linux/Mac: Run ollama serve &
  3. Ensure llama3.2 model is pulled: ollama pull llama3.2
  4. Check port 11434 is not blocked by firewall
  5. Run diagnostic: python test_ollama.py
‚ùå Error: AsyncIO event loop issues ‚ñº

Symptoms:

  • "RuntimeError: This event loop is already running"
  • "Cannot run async function in sync context"

‚úÖ Solution:

Use AsyncClientManager with background thread and asyncio.run_coroutine_threadsafe()

future = asyncio.run_coroutine_threadsafe(
    self._create_client(session_id, model_name),
    self.loop
)
result = future.result(timeout=30)
‚ö†Ô∏è Warning: Module not found ‚ñº

Symptoms:

  • ImportError: No module named 'langchain'
  • ImportError: No module named 'mcp'

‚úÖ Solution:

  1. Activate virtual environment: source venv/bin/activate (Linux/Mac) or venv\Scripts\activate (Windows)
  2. Install dependencies: pip install -r requirements.txt
  3. Verify installation: pip list | grep langchain

üîß Troubleshooting Guide

If Web Server fails:
1. Check if web_server.py is running
2. Verify port 5000 is not in use
3. Check firewall settings
4. Run: python web_server.py
If Ollama fails:
1. Start Ollama: ollama serve
2. Pull model: ollama pull llama3.2
3. Run diagnostic: python test_ollama.py
4. Check port 11434 availability
If RAG API fails:
1. Ensure web server is running
2. Check if /api/rag/query endpoint exists
3. Verify ChromaDB is initialized
4. Check web_server.py has RAG endpoint code
